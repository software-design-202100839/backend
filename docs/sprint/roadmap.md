# 스프린트 로드맵

- **원칙:** 모든 단계가 "왜 지금 이걸 하는가?"에 답할 수 있어야 한다. 이전 단계의 결과물이 다음 단계를 요구할 때만 진행한다.
- **핵심:** 앱이 없으면 인프라가 필요 없고, 배포할 게 없으면 CI/CD가 필요 없다.
- **최종 수정:** 2026-03-27 (Sprint 경계 재편: Sprint 2에서 기능 완결, Sprint 3은 배포+보안만, Sprint 4는 운영만)

## 단계별 "왜 지금?" 근거

### Phase 1 (Sprint 0): 로컬에서 돌아가는 앱 만들기
**근거:** 아무것도 없다. 가장 먼저 필요한 것은 "코드가 로컬에서 실행되는 것"이다.

이 단계에서 하는 것:
- Spring Boot 프로젝트 초기화 → **앱이 있어야 뭐든 할 수 있으니까**
- React 프로젝트 초기화 → **프론트엔드가 있어야 화면이 보이니까**
- ERD + Flyway → **API를 만들려면 DB가 있어야 하니까**
- Docker Compose (PG + Redis) → **로컬 개발환경에서 DB를 띄워야 하니까**
- JWT 인증 → **모든 기능이 로그인 후에 동작하니까, 가장 먼저 필요**
- ESLint/Prettier → **코드를 짜기 시작하니까, 처음부터 규칙을 정해야 나중에 안 싸움**

이 단계에서 하지 않는 것:
- ❌ GitHub Actions CI → 앱이 아직 작으니까 로컬 빌드/테스트로 충분
- ❌ SonarCloud → 분석할 코드가 거의 없음
- ❌ Docker 이미지 빌드 → 배포할 곳이 없음
- ❌ ECS Fargate → 당연히 없음
- ❌ Snyk → 의존성이 아직 적고 안정적인 초기 버전만 사용

### Phase 2 (Sprint 1): 핵심 기능 완성 + 코드 품질 체계 도입
**근거:** 로컬에서 앱이 돌아간다. 이제 핵심 기능을 만들면서 코드가 빠르게 늘어난다. 코드가 늘어나는 시점에 품질 관리를 시작해야 한다.

이 단계에서 하는 것:
- 성적 관리 + 학생부 API/UI → **우선순위 '높음', 핵심 기능**
- GitHub Actions CI 도입 → **여러 페르소나가 PR을 올리기 시작하니까, 자동 검증이 필요해짐**
- SonarCloud + JaCoCo → **코드가 본격적으로 늘어나니까, 이때부터 품질 측정이 의미 있음**
- Snyk 연동 → **의존성이 본격적으로 추가되니까 (Recharts, Spring Security 등)**
- Postman/Postbot API 테스트 → **API가 여러 개 생겼으니까 체계적 테스트 필요**

이 단계에서 하지 않는 것:
- ❌ Docker 이미지 빌드 → 아직 로컬 개발 중, 배포 대상 없음
- ❌ ECS Fargate → 같은 이유
- ❌ Playwright E2E → 화면이 아직 부분적으로만 완성

### Phase 3 (Sprint 2): 전 기능 완성 + 안정화 + 컨테이너화 + 동작 증명
**근거:** 핵심 기능이 완성되고, 나머지 기능도 추가된다. 전 기능이 동작하면 **안정성을 검증하고 증거를 확보**한 뒤, 컨테이너로 패키징한다. Sprint 3에서 배포만 집중하려면, Sprint 2에서 기능과 안정성이 완결되어야 한다.

이 단계에서 하는 것:

| 순서 | 작업 | 담당 | 왜 지금? |
|------|------|------|----------|
| 1 | 피드백 API + UI | 이백엔드, 이프론트 | 우선순위 '중간' |
| 2 | 상담 내역 API + UI | 이백엔드, 이프론트 | 우선순위 '중간' |
| 3 | 알림 (WebSocket + 이메일) | 이백엔드, 이프론트 | 우선순위 '중간' |
| 4 | 학부모 전용 조회 뷰 | 이프론트 | 3-Role 시스템 완성 |
| 5 | **Dockerfile + ECR + CD** (SSCM-51) | 이데브 | Sprint 3 배포의 전제조건, 최우선 |
| 6 | **Playwright E2E + 영상 녹화** (SSCM-52) | 이큐에이 | 동작 증명 = 영상 자동 생성 |

왜 이 시점에 Docker인가:
> "기능이 완성된 앱을 배포하려면 먼저 컨테이너로 패키징해야 합니다. 그래서 Sprint 2에서 Dockerfile을 작성합니다."

왜 이 시점에 Playwright + 영상인가:
> "전 기능이 동작하니까 사용자 시나리오 전체 테스트가 가능합니다. 영상 녹화를 켜면 '이 시스템이 동작한다'는 증거가 자동으로 확보됩니다."

이 단계에서 하지 않는 것:
- ❌ ECS Fargate → Docker 이미지가 방금 만들어졌을 뿐, 클라우드 배포는 다음 단계
- ❌ Prometheus/Grafana/Dynatrace → 운영 환경이 없으니 모니터링할 대상 없음
- ❌ 보고서 생성 → Sprint 3 초반에서 새 프론트엔드 스택과 함께 구현
- ❌ @Version + @Retryable → Sprint 3으로 이월 (Sprint 2는 기능 완결 + 증명에 집중)
- ❌ 상담 필터링 보강 → 기본 CRUD 동작 중, 복합 필터는 Sprint 3
- ❌ Testcontainers → 현재 Mockito 테스트가 동작 중, Sprint 3에서 전환
- ❌ Frontend 기존 코드 리팩터링 → 동작하는 14개 파일 현행 유지 (점진적 마이그레이션)

> **중간고사 (Week 8) 시점:** 전 기능 동작 + E2E 영상 증거 + CI 자동화 + 컨테이너 이미지 준비 완료

### Phase 4 (Sprint 3): 클라우드 배포 + 보안만 집중
**근거:** 컨테이너 이미지가 준비되었다. 이제 "이걸 어디서 돌릴 것인가?"에 답할 차례다. 그리고 실제 서버에 올리기 전에 보안을 점검해야 한다.

**Sprint 3에서는 배포 + 보안에 집중한다.** 작업을 반드시(Must) vs 가능하면(Nice-to-have)으로 분류한다.

### 반드시 (Must) — 배포 + 보안
| 순서 | 작업 | 담당 | 왜 지금? |
|------|------|------|----------|
| 1 | ECS Fargate 클러스터 + 서비스 생성 | 이데브 | Docker 이미지를 실행할 서버리스 환경 |
| 2 | ECS Task Definition 작성 | 이데브 | 컨테이너 실행 설정 (CPU, 메모리, 환경변수) |
| 3 | ALB 설정 (경로 기반 라우팅) | 이데브 | 외부 접속을 위한 HTTPS 진입점 |
| 4 | AWS Parameter Store 설정 | 이데브 | 프로덕션 시크릿 안전 관리 |
| 5 | GitHub Actions CD 완성 (ECR → ECS) | 이데브 | 이미지 빌드 → 배포 자동화 |
| 6 | 개인정보 암호화 (AES-256) | 이백엔드 | 서버에 올리기 전에 보안 적용 필수 |
| 7 | Burp Suite DAST | 이큐에이 | 실제 서버에서 구동 중인 앱 보안 테스트 |
| 8 | 프로덕션 배포 | 이데브 | 스테이징 검증 완료 후 프로덕션 |

### 가능하면 (Nice-to-have) — 남은 기능 + 테스트 강화
| 순서 | 작업 | 담당 | 왜 지금? |
|------|------|------|----------|
| 9 | 보고서 생성 API (OpenPDF + POI) | 이백엔드 | 마지막 남은 기능 |
| 10 | 보고서 UI (새 프론트엔드 스택 적용) | 이프론트 | 신규 코드에만 TanStack Query + RHF + Zod + shadcn/ui 적용 |
| 11 | 상담 필터링 보강 (학생명, 키워드, 복합 필터) | 이백엔드 | 기본 CRUD 위에 고급 검색 추가 |
| 12 | @Version 마이그레이션 + @Retryable (Sprint 2 이월) | 이백엔드 | Optimistic Locking + 이메일 재시도 |
| 13 | Testcontainers 도입 | 이큐에이 | PG + Redis 실제 컨테이너 통합 테스트 |

왜 이 시점에 ECS인가:
> "Sprint 2에서 Docker 이미지가 ECR에 올라갔습니다. 이 이미지를 프로덕션에 배포하려면 컨테이너 실행 환경이 필요합니다. ECS Fargate는 서버 관리 없이 컨테이너만 정의하면 되므로 이 규모에 최적입니다."

왜 EKS가 아닌 ECS Fargate인가:
> "EKS 컨트롤 플레인만 월 $73이고, 클러스터 업그레이드/노드 관리/RBAC 설정 등 운영 오버헤드가 큽니다. SSCM은 모놀리스에 가까운 2~3개 서비스이므로 ECS Fargate(비용 1/5, 운영 부담 1/3)가 적합합니다."

왜 이 시점에 보고서 + 상담 필터링인가:
> "보고서는 우선순위 '낮음'이라 Sprint 2에서 핵심 기능(피드백/상담/알림) 뒤로 미뤘습니다. Sprint 3 초반에 빠르게 처리하고 배포에 집중합니다. 이때 새 프론트엔드 스택을 처음 적용하여 '점진적 마이그레이션' 전략을 실증합니다."

왜 이 시점에 Testcontainers인가:
> "프로덕션 배포 직전에 H2 목 대신 실제 PostgreSQL + Redis 컨테이너로 통합 테스트를 돌려야 프로덕션 환경과의 차이를 잡을 수 있습니다."

### Phase 5 (Sprint 4): 운영 + 모니터링 + 문서화만 집중
**근거:** 앱이 클라우드에 배포되었다. 배포된 시스템을 "운영"하려면 모니터링, 인시던트 관리, 백업이 필요하다.

| 순서 | 작업 | 담당 | 왜 지금? |
|------|------|------|----------|
| 1 | Prometheus + Grafana 설정 | 이데브 | 배포된 앱의 메트릭 수집 + 대시보드 (상시 무료) |
| 2 | Dynatrace APM 연동 | 이데브 | 상용 APM 비교 분석 (15일 체험, 발표 직전 시작) |
| 3 | PagerDuty 연동 | 이데브 | Prometheus AlertManager → 인시던트 알림 체계 |
| 4 | 부하 테스트 (k6) | 이데브 | 실제 서버에서 동시 접속 성능 검증 |
| 5 | **k6 결과 분석 → Redis @Cacheable 도입 판단** | 이백엔드 | DB 병목 확인 시 캐싱 적용, 아니면 스킵 |
| 6 | 백업/복구 절차 | 이데브 | 운영 데이터 보호 |
| 7 | 최종 보안 리포트 | 이큐에이 | 전체 보안 상태 종합 |
| 8 | SonarCloud 최종 점검 | 이큐에이 | 품질 게이트 최종 확인 |
| 9 | 전체 문서 정리 | 전원 | 프로젝트 완료 문서화 |

왜 이 시점에 모니터링인가:
> "Sprint 3에서 프로덕션 배포가 완료되었습니다. 실제 운영 중인 시스템의 성능을 관측하고 장애에 대응하려면 이 시점에서 APM과 인시던트 관리를 도입합니다."

왜 Prometheus + Grafana가 메인이고 Dynatrace가 보조인가:
> "Prometheus + Grafana는 무료이고 Spring Boot Actuator + Micrometer가 네이티브 지원합니다. 상시 운영에 비용 부담이 없습니다. Dynatrace는 상용 APM으로 15일 무료 체험 후 종료되므로 보조로 활용하고, '오픈소스 vs 상용 APM 비교 분석' ADR을 작성하여 프로젝트 깊이를 높입니다."

왜 k6 후에 Redis 캐싱 판단인가:
> "premature optimization을 하지 않습니다. k6 결과에서 DB 조회가 병목으로 확인되면 @Cacheable을 도입하고, 충분하면 스킵합니다. 측정 → 판단 → 적용 순서를 따릅니다."

## Sprint 0 (Week 2~3): 앱 기반 구축 + 인증

**목표:** 로컬에서 프론트엔드 + 백엔드 + DB가 돌아가고, 로그인이 되는 상태

### 하는 것과 이유

| 순서 | 작업 | 담당 | 왜 지금? |
|------|------|------|----------|
| 1 | GitHub 레포 생성 + 문서 커밋 | 이백엔드 | 코드를 저장할 곳이 있어야 시작 가능 |
| 2 | Spring Boot 프로젝트 초기화 | 이백엔드 | 백엔드 API를 만들 기반이 필요 |
| 3 | React+TS+Vite 프로젝트 초기화 | 이프론트 | 프론트엔드 화면을 만들 기반이 필요 |
| 4 | ESLint + Prettier 설정 | 이프론트 | 코드를 짜기 시작하니까 규칙부터 정해야 함 |
| 5 | ERD 설계 + Flyway 마이그레이션 | 이백엔드 | API를 만들려면 DB 테이블이 있어야 함 |
| 6 | Docker Compose (PG + Redis) | 이데브 | 로컬에서 DB/Redis를 띄워야 백엔드가 동작 |
| 7 | JWT 인증 구현 | 이백엔드 | 모든 기능이 로그인 뒤에 동작하므로 첫 번째 기능 |
| 8 | 사용자 Role 구현 | 이백엔드 | 교사/학생/학부모 구분이 있어야 이후 기능 개발 가능 |
| 9 | 로그인/회원가입 UI | 이프론트 | 백엔드 인증 API가 있으니 연동할 화면 필요 |
| 10 | 라우팅 + 인증 가드 | 이프론트 | 로그인이 되니까 보호 페이지 접근 제어 필요 |
| 11 | 인증 API Postman 컬렉션 | 이큐에이 | 인증 API가 완성됐으니 테스트 |

### 하지 않는 것과 이유

| 작업 | 왜 지금 안 하나? | 언제 하나? |
|------|-----------------|-----------|
| GitHub Actions CI | 코드가 적고 PR이 아직 몇 개 안 됨. 로컬 빌드로 충분. | Sprint 1 (코드가 늘어나고 PR이 많아질 때) |
| SonarCloud + JaCoCo | 분석할 코드가 거의 없음. | Sprint 1 (본격적 기능 개발 시작 시) |
| Snyk | 초기 의존성은 안정적 버전. 취약점 스캔 의미 적음. | Sprint 1 (라이브러리 추가될 때) |
| Dockerfile | 배포할 곳이 없으니 이미지를 만들 이유 없음. | Sprint 2 (기능 완성 후 패키징) |
| ECS Fargate | 앱도 없는데 클라우드를 왜 띄우나. | Sprint 3 (이미지 준비 후 배포) |

## Sprint 1 (Week 4~5): 핵심 기능 + 코드 품질 체계

**목표:** 성적/학생부 기능 완성 + CI/품질 도구 도입

**왜 이 시점에 CI/품질 도구?**
> Sprint 0에서 기반이 만들어졌고, 이제 4명이 본격적으로 기능을 개발하면서 PR이 빈번해진다. 코드 품질을 사람이 매번 확인하는 건 비효율적이므로, 자동화된 품질 체계가 필요해진 시점이다.

| 순서 | 작업 | 담당 | 왜 지금? |
|------|------|------|----------|
| 1 | GitHub Actions CI 구성 | 이데브 | PR이 많아지니까 자동 검증 필요 |
| 2 | SonarCloud + JaCoCo 설정 | 이큐에이 | 코드가 본격적으로 늘어나니까 품질 측정 시작 |
| 3 | Snyk 연동 | 이큐에이 | Recharts, Spring Security 등 의존성 추가되니까 |
| 4 | 성적 CRUD API + 자동 계산 | 이백엔드 | 우선순위 '높음' 1번 |
| 5 | 학생부 API | 이백엔드 | 우선순위 '높음' 2번 |
| 6 | 성적 입력 폼 + 레이더 차트 | 이프론트 | 성적 API가 생겼으니 연동 UI |
| 7 | 학생부 조회/수정 UI | 이프론트 | 학생부 API가 생겼으니 연동 UI |
| 8 | 성적/학생부 통합 테스트 | 이큐에이 | API가 완성됐으니 체계적 테스트 |
| 9 | Postbot 자동 테스트 | 이큐에이 | API가 여러 개 생겼으니 엣지 케이스 자동 생성 |
