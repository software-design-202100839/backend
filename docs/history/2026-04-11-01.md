**@Version 낙관적 락 (4개 엔티티):**
- Score, Feedback, Counseling, StudentRecord에 `@Version Long version = 0L` 추가

- `OptimisticLockingFailureException` → 409 `COMMON_004 CONCURRENT_MODIFICATION`로 매핑 (Spring의 기반 클래스를 잡으면 Hibernate `ObjectOptimisticLockingFailureException`도 함께 처리됨)

**@Retryable 이메일 재시도:**
- `RetryableEmailSender` 신규 빈으로 분리 (별도 빈인 이유: spring-retry는 AOP 프록시 기반이라 같은 빈 self-invocation에 적용 안 됨)
- `@Retryable(MailException, max=3, backoff = 1초→2초)` + `@Recover`에서 최종 실패 로그
- `EmailNotificationService`가 `JavaMailSender` 대신 `RetryableEmailSender` 호출
- `RetryConfig`에서 `@EnableRetry`

**테스트 (5건 추가):**
- `RetryableEmailSenderTest` (2건): 3회 재시도 후 @Recover 호출 검증, 정상 발송 시 1회만 호출 검증. spring-retry AOP 프록시 동작 확인을 위해 SpringBootTest로 컨텍스트 부분 로드.
- `GlobalExceptionHandlerTest` (3건): OptimisticLock 2종 + BusinessException 매핑 검증

**최종 테스트:** 111 tests, 0 fail, 1 skip (DB 없는 SscmApplicationTests)

## "왜" 기록

### 왜 SSCM-58을 옵션 A(마지막 날 구현)로 결정했는가
4/9 시점 보류 사유는 "현 규모 동시 충돌 가능성 낮음 + 시간 부족"이었음. 4/11 시점엔:
- 다른 Sprint 4 작업이 모두 끝나 시간 여유가 생김
- 구현 자체가 단순 (엔티티 4 + Flyway 1 + 핸들러 1 + 빈 1, 추산 2~3시간)
- "Sprint 2부터 이월된 이슈를 끝까지 처리"가 프로세스 성실성 근거가 됨
- spring-retry/낙관적 락 패턴 자체가 발표 자료에서 설명할 만한 표준 기법

다만 적용 범위는 신중히 한정:
- `@Retryable`은 알림 발송에만 (사용 빈도 낮음)
- 낙관적 락 충돌은 자동 재시도 없이 사용자 안내 (데이터 정합성 우선)

### 왜 SSCM-8 보고서를 닫지 않았는가
요구사항 문서에 "우선순위 낮음"으로 명시되어 있고, 4주 압축 스프린트에서 의도적으로 스코프 아웃한 기능. 에픽을 Done으로 전환하면 "구현했다"처럼 오해될 수 있음. 오픈 유지 + 스코프 아웃 코멘트가 정직한 기록.

### 왜 SSCM-64를 별도 PR로 분리했는가
SSCM-64(문서)와 SSCM-58(코드)은 서로 다른 관심사 + 다른 페르소나(QA Lee vs Backend Lee). CLAUDE.md "한 번에 하나만" 원칙. 별도 PR로 분리해 리뷰 단위를 작게 유지.
